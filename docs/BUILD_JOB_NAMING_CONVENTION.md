# Build Job Naming Convention

This document describes the Jenkins job naming convention used for OpenJDK build pipeline jobs.

## Overview

Build pipeline jobs follow the same structural naming convention as AQA test jobs, using a `Build_` prefix in place of `Test_`. Jobs are organised across two top-level Jenkins folders:

| Folder | Contents |
|---|---|
| **`OpenJDK_build_launchers`** | One launch orchestrator per active JDK version |
| **`Build_openjdk`** | One platform build job per version/distro/arch/os combination |

## Platform Build Job Name Format

```
Build_openjdk<version>_<distro>_<arch>_<os>
```

| Segment | Description | Example values |
|---|---|---|
| `Build_` | Fixed prefix | — |
| `openjdk<version>` | OpenJDK major version | `openjdk8`, `openjdk11`, `openjdk17`, `openjdk21` |
| `<distro>` | Build distribution — maps to the `VARIANT` pipeline parameter | `temurin`, `dragonwell`, `corretto` |
| `<arch>` | CPU architecture (AQA convention) | `x86-64`, `aarch64`, `ppc64le`, `ppc64`, `s390x`, `aarch32`, `x86-32` |
| `<os>` | Operating system (AQA convention) | `linux`, `mac`, `windows`, `aix`, `solaris` |

## Launch Orchestrator Job Name Format

```
Build_openjdk<version>_launch
```

One per active JDK version, lives in `OpenJDK_build_launchers/`.

## Examples

```
OpenJDK_build_launchers/Build_openjdk21_launch
OpenJDK_build_launchers/Build_openjdk17_launch
OpenJDK_build_launchers/Build_openjdk11_launch

Build_openjdk/Build_openjdk21_temurin_x86-64_linux
Build_openjdk/Build_openjdk21_temurin_aarch64_linux
Build_openjdk/Build_openjdk21_temurin_aarch64_mac
Build_openjdk/Build_openjdk21_temurin_ppc64le_linux
Build_openjdk/Build_openjdk21_temurin_s390x_linux
Build_openjdk/Build_openjdk21_temurin_x86-64_windows
Build_openjdk/Build_openjdk17_temurin_x86-64_linux
Build_openjdk/Build_openjdk11_dragonwell_ppc64le_linux
Build_openjdk/Build_openjdk8_temurin_ppc64_aix
```

## Folder Layout

```
OpenJDK_build_launchers/
├── Build_openjdk21_launch      ← triggers all enabled JDK 21 platforms
├── Build_openjdk17_launch      ← triggers all enabled JDK 17 platforms
└── Build_openjdk11_launch      ← triggers all enabled JDK 11 platforms

Build_openjdk/
├── Build_openjdk21_temurin_x86-64_linux
├── Build_openjdk21_temurin_aarch64_linux
├── Build_openjdk21_temurin_aarch64_mac
├── Build_openjdk21_temurin_ppc64le_linux
├── Build_openjdk21_temurin_s390x_linux
├── Build_openjdk21_temurin_x86-64_windows
├── Build_openjdk17_temurin_x86-64_linux
└── ...
```

## Relation to AQA Test Job Naming

The AQA test suite uses this convention for test jobs:

```
Test_openjdk<version>_<impl|distro>_<testLevel>.<testGroup>_<arch>_<os>
```

Our build jobs mirror the `version`, `distro`, `arch`, and `os` segments with a `Build_` prefix, omitting the test-specific `testLevel`/`testGroup` segments:

```
Build_openjdk<version>_<distro>_<arch>_<os>
```

The `<distro>` segment corresponds to the full distribution name (e.g. `temurin`, `dragonwell`, `corretto`), which AQA also uses in the `<impl>|<distro>` slot alongside the JVM implementation shorthands (`hs`, `j9`). It maps directly to the `VARIANT` pipeline parameter.

## How Job Names Are Derived

Job names are generated by [`ci/jenkins/job-dsl/openjdk_build_pipeline.groovy`](../ci/jenkins/job-dsl/openjdk_build_pipeline.groovy) at job-generation time. The script reads `configurations/jdk<version>_pipeline_config.json` from the config repo and extracts:

| Config field | → | Name segment |
|---|---|---|
| `arch` | → | `<arch>` |
| `os` | → | `<os>` |
| `variant` (or `defaultVariant` from `adoptium_pipeline_config.json`) | → | `<distro>` |

The version number is passed in from the launch job. No manual naming is required.

## Jenkins Views

Two Jenkins views are created automatically by the seed job:

| View | Matches |
|---|---|
| **Build OpenJDK Launchers** | `OpenJDK_build_launchers/Build_openjdk<N>_launch` |
| **Build OpenJDK Platform Pipelines** | `Build_openjdk/Build_openjdk<N>_<distro>_<arch>_<os>` |

## `<arch>` and `<os>` Value Reference

These values come from the `arch` and `os` fields in the platform configuration JSON and must match AQA conventions.

### `<arch>` values

| Value | Description |
|---|---|
| `x86-64` | 64-bit x86 |
| `x86-32` | 32-bit x86 |
| `aarch64` | 64-bit ARM |
| `aarch32` | 32-bit ARM |
| `ppc64le` | 64-bit PowerPC little-endian |
| `ppc64` | 64-bit PowerPC big-endian |
| `s390x` | IBM Z |

### `<os>` values

| Value | Description |
|---|---|
| `linux` | Linux |
| `mac` | macOS |
| `windows` | Windows |
| `aix` | AIX |
| `solaris` | Solaris |
